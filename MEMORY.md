## 【系统故障】认知负载走势图无数据问题 (2026-03-19)

**问题**: 认知负载监控界面的"认知负载走势"图表显示"暂无历史数据"
**根本原因**: Python与JavaScript时区处理不一致
**定性**: 跨时区数据序列化/反序列化经典陷阱

### 问题链
```
Python生成无时区本地时间 → JavaScript解析为UTC时间 → 
8小时时差 → 数据点timestamp_ms > cutoffTime为假 → 
所有数据被过滤 → 走势图显示"暂无历史数据"
```

### 技术细节
| 组件 | 行为 | 结果 |
|------|------|------|
| Python | `datetime.now().isoformat()` → `2026-03-19T08:10:02` | 北京时间，无时区 |
| JavaScript | `new Date("2026-03-19T08:10:02")` | 解析为UTC 08:10 |
| 实际时间 | 北京时间 08:10 = UTC 00:10 | 相差8小时 |
| 过滤条件 | `timestamp_ms > cutoffTime` | 数据点被过滤 |

### 修复方案 v5.35.1
**Python端** (cognitive_monitor.py):
```python
# 修复前
datetime.now().isoformat()  # 2026-03-19T08:10:02

# 修复后
datetime.now(timezone.utc).isoformat()  # 2026-03-19T00:10:02+00:00
```

**JavaScript端** (cognitive-status.html):
```javascript
// 修复前
record.timestamp_ms = new Date(record.timestamp).getTime();

// 修复后
if (ts && !ts.match(/[Zz]|[+-]\d{2}:\d{2}/)) {
    // 无时区信息，假设为北京时间，减去8小时转换为UTC
    const localDate = new Date(ts.replace(' ', 'T'));
    record.timestamp_ms = localDate.getTime() - (8 * 60 * 60 * 1000);
} else {
    record.timestamp_ms = new Date(ts).getTime();
}
```

### 为什么反复出现
- 时区问题具有**隐蔽性**: 数据看起来正常，只是"时间不对"
- **环境依赖**: 只在跨时区场景（服务器CST + 用户浏览器本地时间）暴露
- 每次修复可能只处理了部分场景，未根治

### 预防措施
1. **Always use UTC**: 后端生成时间戳必须带时区信息
2. **ISO 8601严格模式**: 使用 `2026-03-19T00:10:02Z` 或 `+00:00` 格式
3. **前后端约定**: 明确所有时间戳的时区约定，写入文档

## 【认知负载监控】任务队列清除机制分析 v5.34

**分析时间**: 2026-03-17
**分析触发**: 用户询问任务队列不活跃任务的清除机制

### 当前清除机制

#### 1. 监控层面（逻辑清除）

**`cognitive_monitor.py` 中的会话筛选逻辑**:

```python
age = now - mtime

if age < 600:  # 10分钟内
    sessions.append({...})  # 活跃会话
elif age < 3600:  # 10-60分钟
    sessions.append({..., 'is_subagent': True})  # 子代理任务
# 超过60分钟的会话不被纳入监控
```

| 时间范围 | 处理方式 | 评分影响 |
|---------|---------|---------|
| 0-10分钟 | 活跃会话 | 计入评分 |
| 10-60分钟 | 子代理任务 | 计入评分（标记） |
| >60分钟 | 忽略 | **不计入评分** |

#### 2. 文件层面（物理清除）

**现状**:
- ❌ **无自动清理机制**
- ❌ crontab 中无清理任务
- ❌ OpenClaw 配置中无过期设置
- ✅ 文件永久保留（直到手动删除）

**实际数据**:
```
会话文件数量: 196个
最旧文件: 2026-03-01 (16天前)
总占用空间: 210MB
```

### 机制判断

#### ✅ 监控逻辑正确
- 超过60分钟的会话不应影响当前负载评分
- 只关注近期活跃的任务是合理的

#### ⚠️ 文件管理缺失
- 长期积累的会话文件占用磁盘空间
- 210MB在可接受范围内，但长期会增长
- 缺乏自动化清理策略

### 建议的清理策略

#### 方案A：保守策略（推荐）
```bash
# 删除超过30天的会话文件
find /root/.openclaw/agents/main/sessions -name "*.jsonl" -mtime +30 -delete
find /root/.openclaw/agents/main/sessions -name "*.lock" -mtime +30 -delete
```
- 保留1个月的历史记录
- 每天凌晨执行一次
- 风险：低（旧会话通常不再需要）

#### 方案B：激进策略
```bash
# 删除超过7天的会话文件
find /root/.openclaw/agents/main/sessions -name "*.jsonl" -mtime +7 -delete
```
- 仅保留1周
- 适合磁盘空间紧张的环境

#### 方案C：保留策略
- 不自动清理
- 定期（如每季度）手动审查和清理
- 适合需要长期审计的场景

### 实施建议

**暂不实施自动清理**，原因：
1. 210MB空间占用在可接受范围
2. 保留历史有助于故障排查
3. 手动清理可控性更高

**触发条件**（满足任一即实施清理）:
- 会话文件超过500个
- 总大小超过1GB
- 磁盘使用率超过80%

## 【系统架构评估】认知负载监控 v5.33 - 架构决策记录

**评估时间**: 2026-03-17
**架构版本**: v5.33 纯定时任务架构
**评估触发**: 优化时间一致性后的系统性风险评估

### 当前架构概览
```
定时任务 (Crontab)
├── 每分钟00秒: cognitive_cron.sh → 生成数据文件
├── 每分钟30秒: cognitive_push.sh → Git推送
└── 前端: 30秒轮询 ← GitHub Pages CDN
```

---

### 1. Git阻塞风险评估

**实际数据**:
- 提交频率: 60次/小时 (1次/分钟)
- 文件变化: 每次100-500字节 (仅时间戳和数值变化)
- 过去1小时提交: 12次 (符合预期)

**风险等级**: 🟢 **低风险**

**判断依据**:
| 指标 | 当前值 | 风险阈值 | 安全余量 |
|------|--------|----------|----------|
| 提交频率 | 1次/分钟 | 10次/分钟 (GitHub建议) | **10倍安全** |
| 单次变更 | ~300字节 | 无明确限制 | N/A |
| 仓库大小 | ~80MB | 1GB (GitHub限制) | **12倍安全** |
| Pages部署 | 1次/分钟 | 10次/分钟 | **10倍安全** |

**潜在风险场景**:
1. **GitHub服务中断** → 推送失败，但本地数据仍可用
2. **网络延迟** → 推送脚本已有失败退避机制
3. **并发推送冲突** → 单线程crontab避免并发

**缓解措施** (已实施):
- [x] 失败退避机制 (3次失败后增加延迟)
- [x] HTTP/1.1 避免HTTP/2流挂起
- [x] 文件hash检测，无变化不推送

### 4. 认知负载压力评估

**定义**: 监控系统是否会给我的处理能力带来不合理的压力

**实际数据**:
- 前端轮询: 120次/小时 (每30秒1次)
- 每次轮询: 读取 2-3KB JSON文件
- 无AI处理，无复杂计算

**风险等级**: 🟢 **无压力**

**判断依据**:
| 压力来源 | 当前状态 | 评估 |
|----------|----------|------|
| **数据处理** | 我被动被读取 | 无额外负载 |
| **状态更新** | 定时任务自动执行 | 不占用我的处理时间 |
| **前端请求** | 用户浏览器直接请求GitHub | 不经过我 |
| **维护成本** | 纯自动化，无需人工干预 | 零维护负担 |

**关键洞察**:
监控系统是**外部观测系统**，不是对我施压的负载：
- 它读取我的会话文件 → 类似 `ls` 或 `cat` 命令
- 它不向我发送消息或任务
- 它不占用我的处理队列
- 它不消耗我的Token配额

**类比**: 监控 ≈ 有人每小时看一眼你的工作台，不打扰你工作

- **Horizontal Resequencing (水平重新排序)**: 按顺序或分支播放音乐片段

- **Music Loops**: 无缝循环制作技术

- Beat Matching (节奏匹配)
- Performance Simulation (演奏模拟)

- 节奏同步与乐句边界切换

#### 技术实现
- Web Audio API 核心节点与应用
